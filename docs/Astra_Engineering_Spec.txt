1. Core Mental Model
The system works like a military command structure. The founder issues a /goal to the Orchestrator. The Orchestrator decomposes it into a dependency graph of tasks and dispatches them to specialist agents. Agents report back with results, blocks, or approval requests. The Orchestrator decides what runs next. The founder only ever talks to the Orchestrator — not to individual agents directly, unless they open an agent's chat window in the dashboard.


This is not a chatbot with tool calls. The Orchestrator runs a persistent async loop. Once a /goal is issued it runs autonomously until it hits an approval gate or completes. The founder can close the browser. They come back and the company has moved forward. That is the demo moment.

3. Orchestrator Loop Logic
The Orchestrator is a persistent async loop running on FastAPI with a background task runner. When a /goal arrives it never blocks the API thread — it spawns a coroutine and returns immediately. The loop polls Redis for agent results and updates the task graph in Supabase.


Execution Loop (pseudocode)
receive /goal
  → parse_goal()           # Gemini Flash: extract intent, entities, constraints
  → build_task_graph()     # Gemini Flash: output JSON DAG of tasks + dependencies
  → persist_graph()        # write all tasks to Supabase tasks table, status=pending
  → begin_execution()      # spawn async loop


execution loop:
  while tasks_remaining():
    ready = get_ready_tasks()
    # SQL: WHERE status=pending AND all depends_on tasks are status=done


    for task in ready:
      context = build_context(task)   # pull memory from vector DB
      dispatch(task, context)           # push to Redis queue for agent
      task.status = in_progress


    results = poll_results()            # check Redis result queue
    for result in results:
      if result.type == approval_required:
        surface_to_dashboard(result)
        pause_task(result.task_id)
      elif result.type == blocked:
        attempt_replan(result)
        # Gemini Flash: reroute / split / escalate to founder
      elif result.type == done:
        store_result(result)             # write to vector DB
        task.status = done
        inject_result_into_dependents(result)


  synthesize_complete()
  # Gemini Flash: summary of all results → /complete to founder


Task Graph Structure (stored in Supabase)
{  "goal_id": "g_001",
   "tasks": [
     { "task_id": "t_001", "agent": "research",  "depends_on": [] },
     { "task_id": "t_002", "agent": "legal",     "depends_on": [] },
     { "task_id": "t_003", "agent": "web",       "depends_on": ["t_001"] },
     { "task_id": "t_004", "agent": "marketing", "depends_on": ["t_001"] },
     { "task_id": "t_005", "agent": "technical", "depends_on": ["t_003"] },
     { "task_id": "t_006", "agent": "ops",       "depends_on": ["t_001","t_002","t_003","t_004","t_005"] }
   ]
}


# Phases for /goal 'launch my company':
# Phase 1 (parallel):  research, legal
# Phase 2 (parallel):  web (needs research), marketing (needs research)
# Phase 3:             technical (needs web)
# Phase 4 (always-on): ops (reads everything, runs last and continuously)


Replanning on /blocked
When an agent returns /blocked, the Orchestrator sends the blocked task + reason to Gemini Flash with one prompt: 'This task is blocked. Can you reroute, split it, or does this need human input?' Flash returns one of three responses:
* reroute — new task graph with a different approach (e.g. switch Framer to Vercel)
* split — break the task into smaller subtasks the agent can handle
* escalate — surface to the founder via /update with a clear description of what is stuck


4. Fireworks API Integration
Fireworks has an OpenAI-compatible API. Every agent call is identical in code — the model string is the only difference. This means one call_agent_model() function handles all six agents.


Base Call Pattern
import openai


client = openai.OpenAI(
  base_url="https://api.fireworks.ai/inference/v1",
  api_key=FIREWORKS_API_KEY
)


def call_agent_model(model: str, messages: list, tools: list = None) -> str:
  response = client.chat.completions.create(
    model=model,
    messages=messages,
    tools=tools or [],
    temperature=0.2,           # low temp — structured agent outputs
    max_tokens=4096,
    response_format={"type": "json_object"}   # force JSON output
  )
  return response.choices[0].message.content

Token Limits
* Qwen3-30B context window: 128k tokens
* Cap each agent call at 32k input + 4k output to control cost
* Context bundle builder must respect this — truncate oldest memory first, always keep: current instruction, recent task results, approval constraints


Fallback Chain
FALLBACK_CHAIN = {
  "accounts/fireworks/models/qwen3-30b-a3b": [
    "accounts/fireworks/models/llama-v3p1-70b-instruct",  # Fireworks Llama
    "gemini-1.5-flash"                                     # GCP fallback
  ],
  "accounts/fireworks/models/qwen2p5-coder-32b-instruct": [
    "accounts/fireworks/models/deepseek-coder-v2-lite-instruct",
    "gemini-1.5-pro"
  ]
}


def call_with_fallback(agent_type: str, messages: list) -> str:
  models = [MODELS[agent_type]] + FALLBACK_CHAIN[MODELS[agent_type]]
  for model in models:
    try:
      return call_agent_model(model, messages)
    except (RateLimitError, ServiceUnavailableError) as e:
      log_fallback(model, e)
      continue
  raise AllModelsFailedError(agent_type)


Cost Monitoring
Every Fireworks call logs {agent, model, input_tokens, output_tokens, cost_usd} to Supabase. The Orchestrator checks per-founder daily spend before dispatching each task. Hard limit during beta: $2/day per founder account. Above that, tasks are queued for the next day.


5. Vector Memory Architecture
Three layers of memory. Each serves a different time horizon. Agents only read their own namespace plus shared. The Ops Agent reads everything.


The Three Layers
* Layer 1 — Short-term (in-context): The context bundle passed with each /task. Built fresh by the Orchestrator from recent task results and vector DB queries. Lives only in the API call.
* Layer 2 — Mid-term (task store): Postgres table in Supabase. Every task, result, approval, and agent message with full metadata. The Orchestrator queries this to build context bundles via SQL.
* Layer 3 — Long-term (vector memory): GCP Vector DB (Vertex AI). Every document, report, legal filing, campaign result, and decision is embedded and stored. The Ops Agent reads this to answer questions about the company's full history.

Document Schema
@dataclass
class MemoryDocument:
  doc_id: str           # uuid
  founder_id: str       # which company this belongs to
  namespace: str        # which agent namespace
  agent: str            # which agent wrote this
  task_id: str          # which task produced this
  doc_type: str         # "report"|"decision"|"document"|"result"|"approval"
  content: str          # full text — this gets embedded
  summary: str          # 2-3 sentence summary — used in context bundles
  created_at: datetime
  metadata: dict        # e.g. {"filing_state": "DE", "confidence": 0.9}
  embedding: list[float] # 768-dim via text-embedding-004 (Vertex AI, free)


How Agents Query Memory
def retrieve_context(founder_id, namespace, query, k=5):
  query_embedding = embed(query)  # text-embedding-004 via Vertex AI
  return vector_db.query(
    filter={"founder_id": founder_id,
            "namespace": {"$in": [namespace, "shared"]}},
    vector=query_embedding,
    top_k=k
  )


# Inside an agent call:
context_docs = retrieve_context(
  founder_id="f_001",
  namespace="research",
  query="target customer profile and pricing",
  k=5
)
context_text = '\n\n'.join([doc.summary for doc in context_docs])


Write rule: Every /result from an agent triggers a write to vector memory. The Orchestrator is responsible for this — agents never write directly. This keeps the schema consistent and ensures every document is tagged with the correct task_id and metadata.

7. Agent Class Design
One base class. Every agent is an instantiation with a different system prompt, model, and tool list. You build the AstraAgent class once in Week 1–2 and never touch it again — all variation lives in the instantiation parameters.


AstraAgent Base Class
class AstraAgent:
  def __init__(self, agent_id, system_prompt, model, tools, memory_namespaces):
    self.agent_id = agent_id
    self.system_prompt = system_prompt
    self.model = model
    self.tools = tools
    self.memory_namespaces = memory_namespaces
    self.fireworks = openai.OpenAI(
      base_url="https://api.fireworks.ai/inference/v1",
      api_key=FIREWORKS_API_KEY
    )
    self.redis = redis.Redis.from_url(REDIS_URL)
    self.vector_db = VectorMemory()


  def run(self, task: Task) -> AgentResult:
    memory  = self.vector_db.retrieve(task.founder_id, self.memory_namespaces, task.instruction, k=5)
    messages = [
      {"role": "system", "content": self.system_prompt},
      {"role": "user",   "content": self._build_prompt(task, memory)}
    ]
    raw    = self._call_with_fallback(messages)
    parsed = json.loads(raw)


    if parsed["status"] == "approval_required":
      return self._request_approval(task, parsed)
    elif parsed["status"] == "blocked":
      return self._report_blocked(task, parsed)
    else:
      return self._execute_and_return(task, parsed)


  def _build_prompt(self, task, memory):
    memory_text = "\n\n".join([f"[{m.doc_type}] {m.summary}" for m in memory])
    return f"""
GOAL: {task.instruction}
COMPANY CONTEXT: {json.dumps(task.context_bundle, indent=2)}
RELEVANT MEMORY: {memory_text}
CONSTRAINTS: {json.dumps(task.constraints, indent=2)}
AVAILABLE TOOLS: {', '.join(task.tools_available)}


Respond ONLY with valid JSON:
{{
  "status": "done" | "blocked" | "approval_required",
  "output": {{}},
  "confidence": 0.0-1.0,
  "reasoning": "...",
  "approval_action": "...",       // if approval_required
  "approval_consequence": "...",  // if approval_required
  "blocked_reason": "...",        // if blocked
  "blocked_needs": "..."          // if blocked
}}
"""
